# CONTRIBUTING.md

## Как внести вклад

1. **Форкни репозиторий** и создай новую ветку для своей задачи или фикса.
2. **Перед пушем обязательно запускай:**
   - `npm run lint` — проверка стиля кода
   - `npm run test` — запуск тестов
3. **Оформляй коммиты по [Conventional Commits](https://www.conventionalcommits.org/ru/v1.0.0/):**
   - Примеры:
        feat: добавить компонент Sidebar — новая фича
        fix: исправить баг с темизацией — багфикс
        docs: обновить документацию по деплою — изменение документации
        refactor: переписать логику авторизации — рефакторинг без изменения внешнего поведения
        test: добавить тесты для компонента Button — добавление/изменение тестов
        style: исправить форматирование кода — только изменения в стиле (пробелы, точки с запятой и т.д.)
        chore: обновить зависимости — изменения, не влияющие на код приложения (инфраструктура, сборка, зависимости)
        perf: оптимизировать рендеринг списка задач — улучшение производительности
        ci: добавить шаг деплоя в GitLab CI — изменения в CI/CD

        
4. **Описывай Pull Request максимально понятно:**
   - Что сделано, зачем, как тестировать.
   - Если есть breaking changes — обязательно укажи.
   - Связывай PR с задачей через `Closes #номер_issue` (если актуально).
   - Пример описания PR:
     ```
     ## Что сделано
     - Добавлен компонент Sidebar
     - Исправлен баг с темизацией (#42)
     
     ## Как тестировать
     - Перейти на страницу /dashboard
     - Переключить тему, Sidebar должен менять цвет
     
     ## Дополнительно
     - Нет breaking changes
     ```
5. **Добавляй/обновляй тесты** для нового или изменённого функционала.
6. **После ревью и одобрения** — мержи в main/develop (или по правилам проекта).


## Как оформить Issue

### Баг-репорт

- **Заголовок:** Кратко и по делу (например, "Не работает переключение темы в Sidebar")
- **Описание:**  
  - **Шаги для воспроизведения:**  
    1. Открыть страницу /dashboard  
    2. Кликнуть на переключатель темы  
    3. Sidebar не меняет цвет  
  - **Ожидаемый результат:** Sidebar меняет цвет при смене темы  
  - **Фактический результат:** Sidebar всегда светлый  
  - **Скриншоты/логи:** (по возможности приложи)

### Фича/улучшение

- **Заголовок:** Кратко (например, "Добавить drag-and-drop для задач")
- **Описание:**  
  - **Что нужно сделать:** Кратко опиши задачу  
  - **Зачем это нужно:** Обоснуй пользу  
  - **Критерии готовности:**  
    - [ ] Можно перетаскивать задачи мышкой  
    - [ ] Работает на всех страницах  
    - [ ] Есть тесты



## Крупные изменения

- Для крупных изменений (архитектура, новые модули, массовый рефакторинг) — **сначала обсуди идею через Issue или Discussion**.
- Пример:  
  - "Хочу внедрить новую библиотеку для роутинга, потому что ...",  
  - "Планирую рефакторинг слоя features, чтобы ..."


## Чек-лист для Pull Request

- [ ] Код отформатирован и проходит линтинг
- [ ] Все тесты проходят
- [ ] Нет временных/отладочных файлов
- [ ] Нет изменений в package-lock.json без необходимости
- [ ] Нет секретов/ключей в diff
- [ ] Добавлены/обновлены тесты
- [ ] Документация/README обновлены (если нужно)


## Code Style и архитектура

- Используй ESLint и Prettier (см. настройки в проекте).
- Соблюдай архитектурные принципы [Feature-Sliced Design](https://feature-sliced.design/ru/).
- Не коммить `.env` и другие секреты (используй `.env.example`).
- Переиспользуй компоненты и утилиты из shared, не дублируй код.


## Ветвление

- Основные ветки: `main`, `develop`
- Фичи: `feature/имя-фичи`
- Багфиксы: `fix/имя-багфикса`
- Пример: `feature/user-auth`, `fix/navbar-bug`


## Работа с задачами и issue

- Оформляй issue понятно: баг/фича/обсуждение (см. выше)
- Связывай PR с issue через `Closes #номер_issue`


## Рекомендации по ревью

- Оставляй комментарии по делу, конструктивно
- Быстро реагируй на замечания
- Для merge требуется минимум 1 аппрув


## CI/CD

- Все проверки (линтинг, тесты, сборка) проходят автоматически в GitLab CI.
- Не мержи, если pipeline красный.


## Безопасность

- Не коммить реальные ключи и секреты, используй `.env.example`
- О security-issues сообщай через приватный канал или напрямую мейнтейнерам


## Автоматизация

- Используй pre-commit хуки (husky/simple-git-hooks)
- Для автоисправления запускай: `npm run lint -- --fix`


## Инструкции по локальному запуску

- Старт проекта: `npm install && npm run dev`
- Для тестов: `npm run test`
- Для сборки: `npm run build`

Вопросы — через Issues или напрямую мейнтейнерам
