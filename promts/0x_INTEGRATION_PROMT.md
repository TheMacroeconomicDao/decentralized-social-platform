###  Разработка Gybernaty OmniLaunch Matrix (GOL Matrix) — Полнофункциональной экосистемы для запуска R&D-проектов**

**1. Видение (Vision)**
Создать самодостаточную децентрализованную экосистему, где социальный капитал (репутация, экспертиза, организация) и финансовый капитал напрямую конвертируются в запуск новых технологических проектов через механизм сжигания ликвидности. Экосистема состоит из двух стратегических партнеров:
*   **Gybernaty R&D:** Генерирует идеи, экспертизу и исполнение (топливо — **GBR**).
*   **Techhy Venture Club:** Обеспечивает продвижение, сети и венчурные связи (топливо — **VC**).

**2. Ключевые бизнес-требования (Core Business Logic)**
*   **2.1. Управление доступом (GUN — Gybernaty Unit & Access Manager):**
    *   Статусы (`Unit`, `Dev`, `LeadDev`, `ArchDev`, `Core`) присваиваются через **стейкинг чистого GBR**.
    *   Стейкинг работает по принципу **"подушки безопасности"**: пока токены застейканы — статус активен (подтверждается Soulbound NFT, SBT). Вывод стейка мгновенно сжигает SBT и деактивирует доступ к соответствующему функционалу DSP.
    *   Статус `Core` назначается мультисигнатурой совета сообщества.

*   **2.2. Механизм инициализации проектов (Project Ignition & Minting Engine):**
    *   **Тип 1: Внешние проекты (с выпуском токена).**
        1.  Инициатор создает пул ликвидности **GBR / (USDC, DAI, USDT, ETH, BTC, BNB)**.
        2.  LP-токен этого пула безвозвратно сжигается в смарт-контракте `ProjectIgnitionEngine`.
        3.  **В момент сжигания автоматически минтится новый ERC-20 токен проекта** (через фабрику).
        4.  Распределение токенов: **70% — команде проекта**, **20% — в стабилизационный фонд DAO** (управляется держателями статусов от Dev и выше), **10% — резерв (залочены)**.
        5.  Проект автоматически регистрируется в каталоге DSP и может быть выдвинут на продвижение в Techhy VC.
    *   **Тип 2: Внутренние проекты/исследования (без выпуска токена).**
        1.  Для инициирования внутренней инициативы (исследование, обновление DSP) требуется **сжечь определенное количество чистого GBR**.
        2.  Сжигание служит механизмом подтверждения серьезности намерений и дефляции GBR.
        3.  После сжигания команде предоставляется выделенный рабочий пространство в DSP.

*   **2.3. Интеграция с рынком (0x Liquidity & Aggregation Hub):**
    *   **Бесшовный свап и ликвидность:** Интеграция 0x Protocol для агрегации ликвидности по парам **GBR/(USDC, DAI, USDT, ETH, BTC, BNB)** и **GBR/VC**.
    *   **Логика запуска в интерфейсе:** В интерфейсе DEX кнопка "Launch Project" ведет пользователя через процесс: выбор типа проекта -> удобный свап для создания пары (с автоконвертацией при необходимости) -> сжигание LP -> получение токенов.
    *   **Автоматический листинг:** Токены новых проектов (Тип 1) **опционально** могут быть автоматически добавлены в список торговых пар DEX против базового актива их пула (например, PROJX/USDC).

*   **2.4. Decentralized Social Platform (DSP):**
    *   Единый веб-интерфейс, доступ к которому регулируется SBT-статусом (GUN).
    *   Функционал включает: каталог проектов, инструменты коллаборации (чаты, таск-менеджеры, git-репозитории, wiki), дашборды, интеграцию с DEX/Launchpad и 0x.
    *   Экосистема должна быть модульной и расширяемой (10+ потенциальных внутренних сервисов).

**3. Технические требования (Technical Specification)**
*   **3.1. Смарт-контракты (Solidity, аудит обязателен):**
    *   **GUN.sol:** Управление стейкингом GBR, минт/бан SBT статусов.
    *   **ProjectIgnitionEngine.sol:** Основная логика. Должен:
        *   Принимать LP-токены от разрешенных пулов (Uniswap V3, Curve и т.д.).
        *   Валидировать пул (состоит ли он из GBR и разрешенного актива).
        *   Безвозвратно сжигать LP-токен.
        *   Вызывать **TokenFactory.sol** для деплоя нового токена с заданными параметрами и распределением.
        *   Для внутренних проектов — принимать и сжигать чистый GBR.
    *   **TokenFactory.sol:** Фабрика для выпуска стандартизированных, верифицируемых токенов проектов.
    *   **StabilizationFundDAO.sol:** Смарт-контракт казны. Управляется голосованием держателей статусов от Dev и выше. Получает 20% от эмиссии каждого нового токена.
    *   **IntegrationAdapter.sol:** Адаптер для взаимодействия с 0x API и роутерами ликвидности.

*   **3.2. Фронтенд и API:**
    *   **Единый веб-интерфейс (DSP):** React/Next.js + Wagmi/viem. Интеграция с 0x Swap API для агрегации.
    *   **0x Relayer:** Развертывание собственного ретранслятора для обеспечения ликвидности и маршрутизации ордеров.
    *   **Backend for DSP (опционально):** Node.js-сервис для индексации событий с блокчейна (проекты, статусы) и обеспечения работы социальных функций.

**4. Критерии успеха и этапы разработки (Roadmap)**
*   **Фаза 0 (Спецификация):** Детальный аудит существующего кода, проектирование архитектуры всех контрактов, написание полного ТЗ. **Выход:** Document of Understanding и Technical Specification.
*   **Фаза 1 (MVP на тестнете):**
    1.  Разработка и аудит **GUN.sol** и упрощенного **ProjectIgnitionEngine.sol** (только для одного базового актива, например, USDC).
    2.  Базовая интеграция 0x Swap Widget в простой интерфейс.
    3.  Запуск на Sepolia/Ethereum и тестирование полного цикла: стейкинг -> создание пула -> сжигание LP -> получение токена.
*   **Фаза 2 (Расширение MVP):**
    1.  Доработка **ProjectIgnitionEngine.sol** для поддержки всех активов и типов проектов.
    2.  Разработка **TokenFactory.sol** и **StabilizationFundDAO.sol**.
    3.  Создание прототипа DSP с базовым каталогом проектов и проверкой SBT.
*   **Фаза 3 (Полноценный запуск):**
    1.  Полномасштабный аудит всей системы.
    2.  Развертывание на mainnet (Ethereum, затем совместимые L2).
    3.  Развитие функционала DSP, подключение дополнительных инструментов.

**5. Что осталось уточнить для полной готовности?**
1.  **Список "разрешенных активов" для пулов:** Четкий список стейблкоинов и "топ-3" криптовалют. Будет ли он управляемым DAO?
2.  **Параметры распределения для внутренних проектов:** Формула или голосование для определения суммы сжигаемого GBR под конкретную задачу.
3.  **Механика продвижения через VC:** Конкретный смарт-контрактный механизм: как проект "покупает" продвижение за VC токены? Это стейкинг, сжигание или перевод в казну Techhy?
